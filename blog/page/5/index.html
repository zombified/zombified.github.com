Fighting with my system and myself

Inspired by (Manic Digger)[http://manicdigger.sourceforge.net/news/], I decided to do some Mono experimentation.

For my experiments, I decided to use OpenTK -- the same as used in ManicDigger.

I am using monodevelop, but I did not realize it doesn't behave like VS in it's build actions -- it does not copy the assemblies required by your application into it's target directory. So I was getting "The entry point method could not be loaded" error messages, but I could not see the rest of the very helpful message, an issue with the setup I was using. I'm not sure if the message was just being clipped, or what.

After a lot of reconfiguring, poking, and googling, my dumb self finally switched to a different setup and saw the rest of the error message:

** (<project directory>/bin/Debug/bd2cs.exe:7982): WARNING **: The following assembly referenced from <project directory>/bin/Debug/bd2cs.exe could not be loaded:
     Assembly:   OpenTK    (assemblyref_index=0)
     Version:    1.0.0.0
     Public Key: bad199fe84eb3df4
The assembly was not found in the Global Assembly Cache, a path listed in the MONO_PATH environment variable, or in the location of the executing assembly (<project directory>/bin/Debug/).

** (<project directory>/bin/Debug/bd2cs.exe:7982): WARNING **: Could not load file or assembly 'OpenTK, Version=1.0.0.0, Culture=neutral, PublicKeyToken=bad199fe84eb3df4' or one of its dependencies.
The entry point method could not be loaded

The mono runtime can't find the OpenTK.dll assembly. Simple enough -- just setup a command to execute after the project is built to copy the OpenTK.dll file into the target directory!

Ping.1

I've decided that it would be interesting to make a few games and put them out on the Android Market. Nothing fancy, just some fun little arcade games. I'm not planning on porting these games to other systems either, so I can focus on using the Java and the Android SDK. I'm also specifically targetting phones, no tablets or TV's, etc. I'm also only targetting devices that have Android 2.x on them, at least for now.

The first game is going to be called 'Ping' -- not really all that original, but it will suffice. It is going to be a 'Pong' clone, and will not have any gameplay elements other than the original. It's just 2 paddles (AI vs Human), 2 'goals', 2 walls, 1 ball, and a score.

My plan is to make the game as polished as I can -- I want smooth animations, crisp graphics, and as few bugs as possible.

My plan is as follows:

  1. Install and prep Android SDK
  2. Compile sample 'hello world' app and push it to my (dev) phone to make sure everything works
  3. Create a bare-bones app that doesn't do anything except initialize a drawing surface of some kind
  4. Build on the previous app to add input -- input should be reconfigureable by the user (different phones have different inputs). There should be an entirely touch input method.
  5. Build on the previous app to add a background image -- should be reconfigurable by the user
  6. Build on the previous app to add bouncing ball -- there should be no goals yet
  7. Build on the previous app to add 2 paddles -- player should control one of them (which one should be configurable by user)
  8. Build on the previous app to add ball physics to the paddles -- IE, the ball bounces off the paddles
  9. Build on the previous app to add a score-board to the screen
  10. Build on the previous app to remove 2 walls and keep score -- this means tracking when the ball leaves the field, performing a score increment, and reseting the ball
  11. Build on the previous app to make the non-player paddle an ai -- the ai should be configurable as 'easy', 'medium', 'hard', 'impossible'
  12. Clean up the app
  13. Create webpage for it
  14. Publish to Android Market

Definitely not comprhensive, but a good framework for me to work within. I've already done steps 1 and 2. My next entry will be about step 3, creating a bare-bones app that doesn't do anything except initialize a drawing surface of some kind.

A New Host!

I decided to get off of Linode, and the first step is moving my website to a host of some sort. I decided that I didn't need all the features of a CMS or big framework (like drupal, plone, etc, etc), especially with the amount of embeddable web services out there!

Instead, I'm opting to statically generate all my sites (more secure, smaller, and less complex). To do this, I use a really nice application by the name of blogofile. This allows me to template a site, then add content with a somewhat standard formatting language (like markdown).

Another upside to generating a static website is that I can host it pratically anywhere for cheap. The host I ended up picking was Google AppEngine. I picked the AppEngine because it allows me to host a static site (with some fiddling) and allows me to make certain bits of my site dynamic as I need to. It also allows me to host my stuff for free (up to certain quotas, which my site will likely never exceed), and provides a fast distributed system that I don't have to manage at all.

Maybe in a future post I'll go into more detail about my setup, but for now I'm not done yet! I'm just getting started!
