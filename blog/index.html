Creating a Dexterity Type with Dynamic Fields

Currently I'm working on a product to manage events (collective.eventmanager) for Wildcard Corp.. This event manager requires that people register for events -- but each event's registration form may contain different fields.

So, working with Nathan Van Gheem, we figured out a way to make a dexterity type read in a list of field descriptions at runtime, then generate normal add and edit forms.

I have to say, though, that Nathan did most of the figuring -- I was definitely going off on unworkable paths.

Anyway, it turns out it's really quite simple.

The meat of it is in registration.py.

The IRegistration interface defines a few fields that will always be associated with a registration item, and is a regular interface used to define a zope schema for use with dexterity.

To add other fields dynamically, the Edit and Add forms need to be customized, as seen with the EditForm and AddForm classes. In each of the classes, we override the updateFields method, but the call to the addDynamicFields method does most of the work:

def addDynamicFields(form, reg_fields):
    for fielddata in reg_fields:
        field = getattr(schema, fielddata['fieldtype'])(
                    title=unicode(fielddata['name']),
                    required=fielddata['required'])
        field.__name__ = str(fielddata['name'])
        field.interface = IRegistration
        utils.add(form, field)

It, essentially, is taking a list of field definitions. We've chosen to only have a Name, Type, and Required attribute for each field dynamically created, so when we call getattr, we are creating a field based on the through-the-web (dynamic) values.

The field is given a name, and, the most important part, we assign it's interface to be the IRegistration interface, which is used to define the Registration type through the normal dexterity process.

The last line of the funciton calls the utils.add method located in plone.z3cform.fieldsets -- this adds the field to the form and correctly updates everything.

Check out the code, it's pretty slick! Big thank you to Nathan for figuring a bunch of that stuff out!

MelonJS pt 1

Last December I participated in the Ludum Dare (#22), and ultimately used melonJS as my framework of choice.

The next Ludum Dare is coming up, and this time it's the 10th anniversary of the competition!

I need to be prepared.

Therefore, I have decided to do a short series, for my benefit, on how to make a melonJS game.

The game I will be developing is going to be called Minefield, and is based off of the Doger game from Invent With Python.

The full source code will be found on github under my minefield project. Specifically, the code in this post can be found at this commit.

All required files to make this app work are stored in the repository. The libs directory contains all libraries used that aren't specifically created for this project.

There are only 2 significant files (outside of melonJS) in the repo so far:

  1. index.html
  2. js/PlayScreen.js

index.html

This is where most of the magic happens for the base structure needed for the game.

Check out the source, the code is fairly small, and all the important sections have comments.

The one line that I'd like to make a note about is:

<div id="app"></div>

It provides a location for melonJS to create it's render surface. The ID is arbitrary, and is used in the call to "me.video.init".

PlayScreen.js

For now, this is just an empty object. We will be using this object to build up our PLAY state later on.

The Bob Compiler

Check out one of my old college projects I recently dug up: the bob compiler.

My first real big Python program made over the span of a semester in 2007, for a compiler course.

It's hardly commented, horribly designed, and poorly coded, but I'm still proud of how much effor I put into this little project. As I recall we were only ever intended to create a single-pass compiler that simply interpreted tokens as it read them, and spit out assembly as soon as possible.

Instead, I chose to make primitive parser, lexer, semantic analyzer, and translator components. Eventually I was planning on adding optimization options, but I never got to it.
